# Output Compatibility Policy

## Summary

This policy defines what users can depend on when using qml-ng generated components.

## Versioning

- **Generator Version**: Follows semantic versioning (MAJOR.MINOR.PATCH)
- **Schema Version**: Follows MAJOR.MINOR for internal schema structure

See [VERSIONING.md](./VERSIONING.md) for complete details.

## Stability Guarantees

### Will Not Change Without MAJOR Version Bump

These aspects are **stable** and can be relied upon:

- Component class naming: `{Name}Component`
- Component selector: `app-{name}`
- File structure: `{name}.component.{ts,html,scss}`
- Angular standalone component pattern
- Signal and computed() API usage
- TypeScript imports from `@angular/core` and Material modules

### May Change in MINOR Versions

These aspects may evolve in **backward-compatible** ways:

- Internal variable names (e.g., `textExpr1`)
- CSS class names (e.g., `qml-column`)
- Import statement ordering
- Comment formatting
- SCSS default values

### No Guarantees

These may change **at any time**:

- Exact diagnostic messages
- Internal implementation details
- Generated code formatting (whitespace, line breaks)

## What This Means for You

### Safe Practices

✅ **Do**:
- Import generated components in your Angular app
- Use component selectors in templates
- Version control generated files
- Keep original QML sources for regeneration

❌ **Don't**:
- Edit generated files directly
- Depend on internal variable names
- Parse generated code programmatically
- Rely on exact diagnostic text

### When to Regenerate

You should regenerate components when:

1. Upgrading to a new MAJOR version (required)
2. Upgrading to a new MINOR version (recommended, to get improvements)
3. Fixing bugs in your QML source (as needed)

Use the migration schematic for easy regeneration:

```bash
ng generate qml-ng:migrate-qml-component \
  --componentPath=src/app/my-component \
  --qmlFile=path/to/source.qml
```

## Version Metadata

All generated files include version comments for tracking:

```typescript
/**
 * Generated by qml-ng
 * Generator version: 0.3.0
 * Schema version: 1.0
 * Generated at: 2026-04-10T19:39:42.250Z
 */
```

## Deprecation Policy

When we need to make breaking changes:

1. Feature is marked deprecated in version N
2. Deprecation warnings appear in generated code
3. Feature continues to work through version N+1
4. Feature is removed in version N+2

You'll always have at least one MAJOR version to migrate.
