# Versioning and Migration Strategy

This document defines the versioning, compatibility, and migration strategy for qml-ng to ensure stable evolution as the tool matures.

## Overview

Once teams rely on generated Angular components, changing conventions becomes expensive. This strategy provides clear paths for evolving the tool while maintaining backward compatibility where possible and offering clean migration paths when breaking changes are necessary.

## Version Scheme

### Generator Version

The generator version follows semantic versioning (MAJOR.MINOR.PATCH) and is defined in `package.json`:

- **MAJOR**: Breaking changes to CLI interface, schematic options, or output structure that require user intervention
- **MINOR**: New features, additional QML support, or backward-compatible improvements
- **PATCH**: Bug fixes and internal improvements that don't affect generated output

### Schema Version

The intermediate schema version (`SCHEMA_VERSION` in `src/lib/schema/ui-schema.ts`) follows MAJOR.MINOR versioning:

- **MAJOR**: Breaking changes to the schema structure (incompatible changes to `UiDocument`, `UiNode`, etc.)
- **MINOR**: Backward-compatible additions to the schema (new optional fields, new node kinds)

**Current versions:**
- Generator: `0.3.0`
- Schema: `1.0`

### Schema Version Information

The `UiDocument` interface includes structured version information:

```typescript
interface VersionInfo {
  schemaVersion: string;    // e.g., "1.0"
  generatorVersion: string; // e.g., "0.3.0"
  generatedAt: string;      // ISO 8601 timestamp
}
```

## Output Compatibility Policy

### Generated Component Structure

The structure of generated components follows these stability guarantees:

#### Stable (will not change without MAJOR version bump):

- Component class name pattern: `{Name}Component`
- Component selector pattern: `app-{name}`
- File naming: `{name}.component.{ts,html,scss}`
- Angular standalone component pattern
- TypeScript class structure (signals, computed properties)

#### May evolve (MINOR version changes):

- Internal class member names (e.g., `textExpr1` → `text_expr_1`)
- CSS class names for QML types (e.g., `qml-column`)
- Generated import organization
- Comment formatting and diagnostics output

#### Subject to change (any version):

- SCSS default values (gaps, sizes, colors)
- Internal implementation details
- Diagnostic message text

### Schema Evolution

The intermediate `UiDocument` schema may evolve as follows:

#### Backward-compatible changes (MINOR bump):

- Adding new optional fields to existing interfaces
- Adding new node kinds
- Adding new binding types
- Extending `meta` objects with new properties
- Adding deprecation markers

#### Breaking changes (MAJOR bump):

- Removing existing fields
- Changing field types
- Restructuring node hierarchies
- Renaming interfaces or enums
- Changing required fields

## Migration Strategy

### When Migration is Needed

Migration is needed when:

1. The schema MAJOR version changes
2. The generator introduces breaking changes to output conventions
3. New features require updates to existing components
4. Deprecated features are removed

### Migration Schematic

Use the `migrate-qml-component` schematic to update existing generated components:

```bash
ng generate qml-ng:migrate-qml-component \
  --componentPath=src/app/login \
  --qmlFile=examples/login.qml
```

#### Options:

- `--componentPath`: Path to the existing generated component directory
- `--qmlFile`: Path to the original QML source file
- `--force`: Force regeneration even if schema version is current

#### Migration Process:

1. The schematic reads the existing component's version metadata
2. Compares the schema version with the current version
3. If migration is needed (or `--force` is specified):
   - Regenerates the component from the QML source
   - Overwrites all three files (TS, HTML, SCSS)
   - Preserves the directory structure
4. Logs diagnostics and migration status

### Manual Migration

If the QML source file is not available:

1. Extract the version metadata from the existing component
2. Consult the [Migration Guide](#migration-guides) for your version
3. Manually update the component following the migration steps
4. Update the version comments in the file headers

## Deprecation Policy

### Marking Features as Deprecated

Features that will be removed or changed should be marked as deprecated:

```typescript
interface DeprecationInfo {
  feature: string;         // e.g., "anchors.fill approximation"
  reason: string;          // Why it's deprecated
  alternative?: string;    // Recommended replacement
  removeInVersion?: string; // Version when it will be removed
}
```

Deprecations can be attached to:
- Individual nodes (`UiNode.deprecations`)
- The entire document (`UiDocument.deprecations`)

### Deprecation Lifecycle

1. **Announce**: Add deprecation info to schema and diagnostics
2. **Grace Period**: Maintain deprecated feature for at least one MAJOR version
3. **Warn**: Emit warnings in generated code comments
4. **Remove**: Remove in announced version with clear migration path

### Example Deprecation

When a layout pattern is being replaced:

```typescript
{
  feature: "anchors.fill exact matching",
  reason: "CSS Flexbox provides better cross-browser layout fidelity",
  alternative: "Use explicit width: 100% and height: 100% with flex layout",
  removeInVersion: "2.0"
}
```

## Bundle Conversion and Multi-File Projects

As the tool evolves to support multi-file QML projects:

### Versioning for Bundles

- Each component file maintains its own version metadata
- Bundle-level conversion adds a manifest file tracking all generated components
- Migration schematic can regenerate entire bundles

### Asset Handling

- Asset references (images, fonts, shaders) are tracked in version metadata
- Asset migration paths are documented per version
- Breaking changes to asset handling trigger MAJOR version bump

## Validation and Compatibility Checks

### Schema Validation

The converter validates schema versions at runtime:

```typescript
export function qmlToUiDocument(name: string, qml: QmlDocument): UiDocument {
  // ... conversion logic ...

  const version: VersionInfo = {
    schemaVersion: SCHEMA_VERSION,
    generatorVersion: packageVersion,
    generatedAt: new Date().toISOString()
  };

  return { name, root, diagnostics, version };
}
```

### Migration Detection

The migration schematic automatically detects when migration is needed:

```typescript
function needsMigration(metadata: VersionMetadata | null): boolean {
  if (!metadata || !metadata.schemaVersion) {
    return true; // No version metadata, assume migration needed
  }

  const currentMajor = parseInt(SCHEMA_VERSION.split('.')[0], 10);
  const existingMajor = parseInt(metadata.schemaVersion.split('.')[0], 10);

  return existingMajor < currentMajor ||
         metadata.schemaVersion !== SCHEMA_VERSION;
}
```

## Best Practices for Users

### Development Workflow

1. **Keep QML sources**: Always maintain the original QML files for regeneration
2. **Don't edit generated files**: Treat generated components as build artifacts
3. **Version control both**: Commit both QML sources and generated components
4. **Regular regeneration**: Periodically regenerate to pick up improvements
5. **Test after migration**: Run your Angular test suite after migrating components

### When Updating qml-ng

1. Review the CHANGELOG for breaking changes
2. Run the migration schematic on test components first
3. Validate output in a staging environment
4. Update all components before deploying to production

### Custom Extensions

If you need to customize generated components:

1. **Wrapper pattern**: Create wrapper components that import generated ones
2. **Composition**: Use Angular composition patterns rather than editing generated code
3. **Feature requests**: Request enhancements to the generator instead of patching output

## Stability Guarantees

### What We Guarantee

- **Schema compatibility**: MINOR versions are backward-compatible
- **Migration path**: Breaking changes include migration tools
- **Deprecation notice**: At least one MAJOR version notice before removal
- **Version tracking**: All generated code includes version metadata

### What We Don't Guarantee

- **Exact output**: Internal implementation may change
- **Performance**: Optimization may affect code structure
- **Diagnostics**: Warning messages may be refined
- **CSS specifics**: Default styles may evolve
